How It Works

Understanding WolfScale's core mechanisms

Write Flow

  1. Client sends write to any node
  2. If follower, request forwarded to leader
  3. Leader logs to WAL and replicates
  4. Followers apply and acknowledge
  5. Leader confirms the write

WAL (Write-Ahead Log)

Batching Groups operations for efficiency
Compression LZ4 reduces storage/network
Segmentation 64MB segment files
Retention Configurable (default 7 days)
Durability Optional fsync

Deterministic Leader Election

The node with the lowest ID among active, synced nodes becomes leader.

Example: If node-1 fails, node-2 becomes leader (node-2 < node-3)
No split-brain Only one lowest ID
Predictable failover Know who becomes leader
Instant transition No voting rounds

Node Recovery

  1. Returns as Follower
  2. Reports last LSN to leader
  3. Leader sends missing WAL entries
  4. Node replays in order
  5. Becomes Active when caught up
Important: A node must be synced before becoming leader.

MySQL Proxy Mode

Connect via MySQL
mysql -h any-node -P 3307 -u user -p

Query Routing

INSERT/UPDATE/DELETE/DDL → Leader
SELECT on leader → Local DB
SELECT on synced follower → Local DB
SELECT on lagging follower → Leader

Cluster Communication

heartbeat_interval_ms 500ms default
Heartbeat timeout 3x interval
election_timeout_ms 1500-3000ms
