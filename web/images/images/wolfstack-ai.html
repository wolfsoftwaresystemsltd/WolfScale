ü§ñ AI Agent

Built-in AI assistant that can answer questions and run commands across your cluster

Overview

Every WolfStack node includes a built-in AI operations assistant powered by Claude (Anthropic) or Gemini (Google). The AI agent lives in the bottom-right corner of the dashboard as a pulsing "AI" button ‚Äî click it to open a chat panel and ask anything about your servers.

What makes WolfStack's AI different is that it doesn't just tell you commands to run ‚Äî it runs them for you, locally or across your entire cluster, and presents a clear summary of the results. Need to know what CPU every server runs? Ask ‚Äî the AI will query every node and report back.

üõ°Ô∏è Safety First: The AI can only execute read-only commands. It cannot delete files, stop services, install software, or make any destructive changes.

üí¨ What It Can Do

Ask Questions

Ask about WolfStack, WolfNet, WolfDisk, WolfProxy, WolfServe, or WolfScale. The AI has the full documentation loaded as its knowledge base.

Run Local Commands

The AI can run read-only commands on the server you're connected to. Ask "what's our disk usage?" and it'll run df -h and summarize.

Run Cluster Commands

Ask about all your servers and the AI will execute commands across every online node in your cluster and present a combined report.

Health Monitoring

Background health checks run on a configurable interval. When issues are detected, the AI can send email alerts and log them to the alert history.

Troubleshooting

Describe a problem and the AI will investigate ‚Äî checking processes, logs, network status, and container health to help diagnose the issue.

Update Checking

The AI knows the current WolfStack version and can check GitHub for newer releases, so you always know if an update is available.

‚öôÔ∏è How Command Execution Works

When you ask the AI a question that requires live data, it follows a multi-turn execution loop:

  1. You type a question like "What CPUs are we running across the cluster?"
  2. WolfStack sends your message to the LLM (Claude or Gemini)
  3. The AI responds with command tags: [EXEC_ALL]lscpu[/EXEC_ALL]
  4. WolfStack executes lscpu on the local server
  5. WolfStack proxies the command to every online cluster node via /api/ai/exec
  6. All command output is collected and sent back to the AI
  7. The AI analyses the results and presents a clear human-readable summary

There are two command modes:

Tag Scope Example
[EXEC]‚Ä¶[/EXEC] This server only [EXEC]df -h[/EXEC]
[EXEC_ALL]‚Ä¶[/EXEC_ALL] All cluster nodes [EXEC_ALL]free -h[/EXEC_ALL]

The AI can request up to 3 rounds of command execution per question to prevent infinite loops. Each round is analysed before deciding whether more data is needed.

üõ°Ô∏è Safety & Restrictions

Command execution is locked down with a defence-in-depth approach. The AI can only run commands that pass all of the following checks:

‚úÖ Allowed Commands (Whitelist)

Only these command families are permitted:

Category Commands
System info uname hostname uptime lscpu lsblk lsmem free df du arch nproc
Proc info cat /proc/cpuinfo cat /proc/meminfo cat /etc/os-release cat /etc/hostname
Processes ps top -bn1 systemctl status systemctl list-units journalctl
Network ip addr ip route ss netstat ping -c dig nslookup traceroute
Containers docker ps docker stats docker inspect docker logs docker images lxc-ls lxc-info
Files (read) ls cat head tail wc find stat
Misc date env timedatectl hostnamectl sensors smartctl

üö´ Blocked Commands

The following are always blocked, even if piped through an allowed command:

Category Blocked
Destructive rm rmdir unlink dd mkfs fdisk parted
System control shutdown reboot poweroff halt kill killall pkill
File modification mv cp chmod chown nano vim sed -i tee
User management useradd userdel passwd visudo
Service control systemctl start/stop/restart/enable/disable
Package managers apt yum dnf pip npm cargo
Container ops docker rm/rmi/stop/kill/exec/run/pull/push/build
Firewall iptables (modify rules), nft, firewall-cmd

üîí Additional Safeguards

  • No output redirection ‚Äî >, >>, and 2> are blocked
  • No command substitution ‚Äî backticks (`) and $() are blocked
  • Pipe safety ‚Äî every segment of a piped command is checked against the blocklist
  • Output limit ‚Äî command output is truncated at 10,000 characters
  • Timeout ‚Äî commands that take longer than 10 seconds are terminated
  • Max rounds ‚Äî the AI is limited to 3 command rounds per question
  • Authentication required ‚Äî all endpoints require a valid WolfStack session

üîß Configuration

Open the AI chat panel and click the ‚öôÔ∏è Settings button to configure:

Setting Description
Provider Choose between Claude (Anthropic) or Gemini (Google)
API Key Your API key for the chosen provider. Stored securely on the server at /etc/wolfstack/ai-config.json
Model The model list is fetched dynamically from the provider's API. Select your preferred model.
Email Alerts Enable SMTP email alerts for health check findings. Configure your SMTP server, port, username, password, and recipient address.
Check Interval How often the background health check runs (default: 60 minutes)

{
  "provider": "claude",

  "email_enabled": true,
  "email_to": "ops@example.com",
  "smtp_host": "smtp.gmail.com",
  "smtp_port": 587,
  "smtp_user": "alerts@example.com",
  "smtp_pass": "app-password",
  "check_interval_minutes": 60
}

üì° API Endpoints

Method Endpoint Description
GET /api/ai/config Get AI config (API keys masked)
POST /api/ai/config Save AI config (provider, keys, model, email settings)
POST /api/ai/chat Send a chat message ‚Äî triggers multi-turn command execution
POST /api/ai/exec Execute a safe read-only command (used by cluster proxy)
GET /api/ai/status AI agent status (configured, provider, model, alerts)
GET /api/ai/alerts Historical health check alerts
GET /api/ai/models List available models for the configured provider
