üì° API Reference

Complete REST API & WebSocket Documentation for WolfStack

Overview

WolfStack exposes a RESTful JSON API on port 8553 (default). All endpoints require session authentication (cookie-based) unless otherwise noted.

Authentication

Log in with your Linux system credentials to receive a session cookie. All subsequent requests must include this cookie. Sessions expire after 8 hours.

WebSocket Console

Interactive terminal access via WebSocket:

url 
ws://localhost:8553/ws/console/{type}/{name}
  • type: docker, lxc, or host
  • name: Container name (for docker/lxc) or ignored (for host)
bash 
# Login (returns Set-Cookie header)
curl -c cookies.txt -X POST http://localhost:8553/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username": "root", "password": "your-password"}'

# Use the session cookie for subsequent requests
curl -b cookies.txt http://localhost:8553/api/metrics
üîê PAM Authentication: WolfStack authenticates against your Linux system users via PAM. Any user in /etc/shadow can log in. WolfStack requires root to read shadow passwords.

üíæ Storage Manager

Mount and manage S3, NFS, directory, and WolfDisk storage from the dashboard. S3 mounts use rust-s3 (pure Rust ‚Äî works on IBM Power/ppc64le, no C/C++ deps) as the primary mount method, with s3fs-fuse as a fallback.

‚òÅÔ∏è Supported S3 Providers: AWS S3, Cloudflare R2, MinIO, Wasabi, DigitalOcean Spaces, Backblaze B2, WolfDisk S3 ‚Äî any S3-compatible endpoint.
Method Endpoint Description
GET /api/storage/mounts List all storage mounts with live status (mounted/unmounted/error)
GET /api/storage/available List mounted storage suitable for container attachment
POST /api/storage/mounts Create a new S3, NFS, directory, or WolfDisk mount
PUT /api/storage/mounts/{id} Update mount configuration
DELETE /api/storage/mounts/{id} Remove a mount entry (unmounts first if mounted)
POST /api/storage/mounts/{id}/mount Mount a storage entry
POST /api/storage/mounts/{id}/unmount Unmount a storage entry
POST /api/storage/mounts/{id}/sync Sync mount config to all cluster nodes (global mounts)
POST /api/storage/mounts/{id}/sync-s3 Push local changes back to S3 bucket (rust-s3 upload)
POST /api/storage/import-rclone Import S3 remotes from rclone.conf content

POST /api/storage/mounts ‚Äî Create S3 Mount

Request Body

  "name": "my-r2-bucket",
  "type": "s3",

  "global": true,
  "auto_mount": true,
  "s3_config": {
    "bucket": "my-bucket",
    "provider": "Cloudflare",
    "region": "auto",
    "endpoint": "https://abc123.r2.cloudflarestorage.com",
    "access_key_id": "AKID...",

}

POST /api/storage/import-rclone

Import existing rclone.conf configurations as WolfStack S3 mounts:

Request Body
{
  "config": "[my-r2]\ntype = s3\nprovider = Cloudflare\naccess_key_id = AKID...\nsecret_access_key = SECRET...\nendpoint = https://abc.r2.cloudflarestorage.com\n"
}
Response (200 OK)
{
  "message": "Imported 1 S3 remotes",
  "mounts": [
    {
      "id": "abc123",
      "name": "my-r2",
      "mount_type": "s3",
      "mount_point": "/mnt/wolfstack/my-r2",
      "status": "unmounted",
      "s3_config": { "bucket": "", "provider": "Cloudflare", ... }
    }
  ]
}

POST /api/storage/mounts/{id}/sync-s3

Push local changes back to the S3 bucket. Uses rust-s3 to upload files from the local cache directory (/var/cache/wolfstack/s3/{id}/) to the bucket:

Response (200 OK)
{
  "message": "Synced 42 files to S3"
}
ü¶Ä How rust-s3 Mounting Works: WolfStack uses the rust-s3 crate to list and download bucket objects to a local cache directory (/var/cache/wolfstack/s3/{mount-id}/), then bind-mounts it to the chosen path. This avoids FUSE-over-network (rclone/s3fs) overhead and works on any architecture including IBM Power (ppc64le). Files are synced on mount and can be pushed back with sync-s3.

üíª WebSocket Console

Protocol Endpoint Description
WS /ws/console/{type}/{name} Interactive terminal ‚Äî full PTY to Docker or LXC container

Parameters

  • {type} ‚Äî docker or lxc
  • {name} ‚Äî Container name or ID

Connection

JavaScript
// Connect to a Docker container's shell
const ws = new WebSocket('ws://localhost:8553/ws/console/docker/my-nginx');

// Send input
ws.send('ls -la\n');

// Receive output
ws.onmessage = (event) => {
  terminal.write(event.data);
};
üîß Under the hood: The server spawns a PTY and runs docker exec -i or lxc-attach. Shell detection prefers /bin/bash if available, otherwise falls back to /bin/sh. The WebSocket carries raw terminal bytes in both directions.
